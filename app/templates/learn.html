How it all works

Every concept this tool uses, explained from scratch — including where each retrieval strategy tends to break. Read top to bottom, or jump to what you're curious about.

The basics

What is a token?
Models don't read letters or words — they read tokens, chunks of text a tokenizer carved out by frequency. Common words are a single token; rarer words split into pieces (unbelievable → un + believ + able). Even a leading space changes the token. Token counts — not word counts — are what drive model cost and context limits.
What is an embedding model?
It turns a piece of text into a fixed-length list of numbers (a vector) that captures its meaning. Two texts with similar meaning land close together in that space, even if they share no words. The vector length is fixed per model (e.g. 768) no matter how long the input is — open the 🦙 Ollama panel on the main page and inspect a model's embedding_length to see it.
What is a vector database / similarity search?
Once every chunk of your document is a vector, "finding relevant text" becomes "finding the nearest vectors to the question's vector." Closeness is measured with cosine similarity (1.0 = same direction/meaning, 0 = unrelated). This tool computes that directly in Python; a production system would use a dedicated vector database with a fast approximate-nearest-neighbor index.
Chunking & overlap — why?
Embedding models have a maximum input size, and huge chunks blur meaning while tiny ones lose context. So we split text into chunks. Overlap shares a little text between neighboring chunks so a sentence sitting on a boundary isn't cut in half.
Gotcha: too much overlap makes neighboring chunks near-duplicates, which can confuse diversity-aware strategies like MMR — a real interaction between two settings that each seem reasonable alone.
What is RAG?
Retrieval-Augmented Generation. Instead of relying only on what a model memorized in training, you retrieve relevant chunks of your documents at question time and hand them to the model as context before it answers. That lets it answer about private notes, a codebase, or documents that didn't exist when it was trained — and, done well, refuse to guess when the answer isn't there.

The retrieval strategies

Top-K
The baseline: embed the question, return the k chunks whose vectors are closest. Simple and usually strong. Every other strategy is measured against it.
MMR (diversity-aware)
Like Top-K, but after the first pick it penalizes chunks too similar to ones already chosen — trading some relevance for variety.
It can backfire: if two adjacent chunks are both correct but look alike, MMR may drop one and hurt accuracy. That's exactly why this tool measures hit rate instead of assuming MMR is an improvement.
Hybrid (meaning + keywords)
Combines vector search with classic keyword search (BM25) and fuses the two rankings with Reciprocal Rank Fusion. Vectors are great at meaning but can miss an exact term — a product code, a name — that keyword search nails instantly. Hybrid catches both.
HyDE
Asks a chat model to first write a hypothetical answer to your question, then searches using that answer's embedding instead of the question's. Answer-shaped text often matches document-shaped text better than a question does. Costs one extra LLM call per question.
Contextual Retrieval
Before embedding, a chat model writes a 1–2 sentence blurb saying what each chunk is about, and prepends it. So a lonely chunk like “it dropped 15% that quarter” gets embedded together with “this is about ACME's Q3 2023 revenue.” A big retrieval boost — but one LLM call per chunk makes indexing slow.

Reading the results

What do the metrics mean?
Relevance: how close the retrieved chunks are to the question (higher is better). Redundancy: how similar the retrieved chunks are to each other (lower = more varied coverage). Hit Rate: for questions with an expected keyword, how often that keyword actually showed up in the retrieved chunks — the only true accuracy metric here. Latency: retrieval time.
High relevance but low hit rate — what does that mean?
It means the strategy confidently retrieved chunks that look related to the question but don't actually contain the answer. This is the single most useful thing to catch before building a real pipeline, and why a keyword-based hit rate is worth the effort of supplying ground truth.
Hit rate is 0% but the generated answer was correct — is it broken?
No — those measure two different steps. Hit rate checks the retrieval step: did a chunk containing your literal keyword get pulled? The grounded answer checks the generation step: did the model answer well from whatever was retrieved. If retrieval pulled the right chunk but your keyword wasn't a literal text match, you get a MISS and a correct answer at the same time — the keyword check is a crude proxy, not the source of truth.
Keywords are split on commas, so Django, FastAPI counts as two acceptable keywords and matches text like "Django and FastAPI". Expand a result row to see exactly which keywords were searched and which matched.
Why does the first run feel slow?
Ollama loads a model into memory the first time you use it, which can take several seconds. After that it stays warm and later calls are much faster. Contextual Retrieval and HyDE are slower by nature too, because they make extra chat-model calls.
How accurate is this tool, really?
It's a rough, local first read — not a benchmark. Chunk token counts use an approximate tokenizer, everything is held in memory, and search is brute-force. The goal is to build intuition and catch obvious problems (bad chunk size, a weak embedding model, a strategy that hurts) before you invest in a full pipeline.
